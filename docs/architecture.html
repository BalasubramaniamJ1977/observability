Payment Observability — UETR Tracer

End-to-end distributed tracing, metrics, and structured logging for an 8-service synchronous payment pipeline, correlated by UETR across every stage from initiation to network dispatch.

Python 3.11 · FastAPI OpenTelemetry SDK 1.22 Tempo · Prometheus · Loki Grafana 13 8 Microservices UETR = Trace ID

Overview

This project instruments a realistic payment-processing pipeline so that any payment can be traced from the moment a client submits it to the moment the external network acknowledges it — using a single UETR (Unique End-to-End Transaction Reference, a UUID) as the correlation key across traces, metrics, and logs.

Distributed Tracing

Every HTTP hop between the 8 services produces a child span that carries the payment UETR as a span attribute. Grafana's Tempo explorer shows the full flame graph for any payment in one view.

Service Metrics

Each service exposes request rate, error rate, and processing latency as OTLP metrics. Prometheus scrapes a collector endpoint. Grafana panels show per-service P99/P50, error rates, and throughput (req/s, payments/min).

Structured Logs

Python's standard logging module is bridged to the OTEL LoggerProvider. Logs are shipped to Loki with payment-specific fields (UETR, amount, currency, rail) and are filterable by the same UETR textbox that drives the trace panel.

UETR ↔ Response-UETR Linkage

The final service (response-handler) generates a new UETR for the network ACK and records both UETRs on the same span via payment.original_uetr, payment.new_uetr, and a uetr.linked span event.

All signals (traces, metrics, logs) flow through a single OTEL Collector before fan-out to their respective backends. This means services have only one endpoint to configure (OTEL_EXPORTER_OTLP_ENDPOINT) and the collector handles fan-out, batching, and backpressure.

Payment Pipeline

The pipeline is fully synchronous — each service awaits its downstream HTTP call before returning, so the trace flame graph reflects real end-to-end wall-clock latency at each stage. Response flows back through the same chain in reverse.

Key Design: UETR = Trace ID

⚡

The most important design decision in this project: the payment-initiation service forces the OTEL trace_id to equal the UETR (UUID). Because a UUID is a 128-bit number and so is a Tempo trace ID, this is a lossless mapping. All downstream services inherit the same trace_id via W3C traceparent propagation, so the trace ID and the UETR are the same identifier — you can find the trace in Tempo either way.

How It Works

def _uetr_to_trace_id(uetr: str) -> int:
    # UUID (with dashes) → 128-bit int used as OTEL trace_id
    return int(uetr.replace("-", ""), 16)

parent_ctx = trace.set_span_in_context(
    NonRecordingSpan(
        SpanContext(
            trace_id=_uetr_to_trace_id(payment.uetr),
            span_id=random.getrandbits(64),
            is_remote=True,
            trace_flags=TraceFlags(TraceFlags.SAMPLED),
        )
    )
)

with tracer.start_as_current_span("payment.initiate", context=parent_ctx, ...) as span:
    ...
Benefits

Any operator who has the UETR from a payment report can paste it directly into the Grafana UETR textbox or into Tempo's trace ID search and reach the same trace — no lookup table required.

Response UETR Linkage

response-handler generates a new UUID as the network ACK UETR. Both UETRs are stored on the same span as payment.original_uetr and payment.new_uetr, plus a uetr.linked span event that is visible in the Tempo flame graph.

Key Design: Automatic Trace Propagation

No service manually injects or extracts W3C headers. The two auto-instrumentors handle everything:

DirectionInstrumentorWhat it does
Inbound FastAPIInstrumentor Extracts traceparent from the incoming HTTP request and makes the parent span available as the current context. Creates a server span automatically.
Outbound HTTPXClientInstrumentor Injects traceparent (and tracestate) into every outbound httpx.AsyncClient request. Creates a client span that is a child of the current span.
ℹ️

Each service's business span (e.g. routing.select_rail) is created with start_as_current_span(…) inside the server span that FastAPI already created. The httpx call therefore carries the business span as its parent, not the raw server span — giving a clean three-level hierarchy: server span → business span → httpx client span.

Key Design: Dynamic UETR Search in Grafana

Both the trace panel and the logs panel use the same $uetr dashboard variable with anchored regex matching:

Tempo — TraceQL
{ span.payment.uetr =~ "^${uetr}$" }
Loki — LogQL
{service_name=~".+"} | payment_uetr =~ "^${uetr}$"

Behaviour by Variable Value

Variable valueTraceQL expands toResult
.* { span.payment.uetr =~ "^.*$" } Matches any UETR → shows 20 most recent traces
8a562952-d7f2-… { span.payment.uetr =~ "^8a562952-d7f2-…$" } Anchored regex → exact match → only that payment's trace
✓

The ^…$ anchors are critical: without them =~ is a contains-match (substring match), which could return false positives if one UETR happens to be a substring of another string. With anchors the semantics match = (exact) when a full UETR is entered, and "show all" when the wildcard default is used.
